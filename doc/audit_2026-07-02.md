## Problems (severity-ranked, with fixes)

### Critical

**1. Gemini tool-calling is silently broken since the SDK migration**
`mcp/gateway.py:~1249` only attaches `tool_calls` to the assistant message when `message_format == 'openai'` — no Gemini branch, so the model's own function call is dropped from history. Separately, `_append_tool_result` (`gateway.py:~1976-1983`) still emits the old pre-migration REST shape for Gemini (`role:'user', parts:[{functionResponse:...}]`), but the new SDK adapter (`mcp/providers/gemini.py::_build_contents`) only recognizes `role:'tool'` for results and a `tool_calls` list on assistant messages. Neither shape matches what gateway.py actually sends. **Every tool call in a Gemini conversation silently degrades to an empty turn** — the model never sees its own function call or the tool's result.
Went uncaught because the one regression test for it is itself broken (see #7).
*Fix:* add a `message_format == 'gemini'` branch at `gateway.py:1249` mirroring the OpenAI one (attach `tool_calls` as `[{name, arguments}]`); change the gemini branch in `_append_tool_result` (line 1976) to `{'role': 'tool', 'tool_call_id': ..., 'name': tool_name, 'content': result}`.

**2. SSRF via `_dispatch_set_binary_field`**
`mcp/tools/dispatcher.py:459` fetches an AI-supplied URL via `urllib.request.urlopen` with no scheme/host validation, then writes the response into a binary field the AI can read back. A prompt-injected model could direct the Odoo server to hit internal network addresses or cloud metadata endpoints (`169.254.169.254`) and exfiltrate the response through the field.
*Fix:* restrict to `http(s)`, resolve the hostname and reject loopback/link-local/RFC1918 ranges before fetching.

### High

**3. XSS-shaped gap — unsanitized raw HTML render**
`static/src/xml/mcp_chat_widget.xml:419` — the `_type === 'html'` branch does `<t t-out="message.structuredData.content"/>` with no sanitization, unlike every other content path (`t-esc`, or `t-out` through the markdown renderer). `structuredData` comes directly from the LLM's JSON reply, so a manipulated model response could inject `<img onerror=...>` and execute in the chat UI.
*Fix:* sanitize server-side (`html_sanitize`) before this reaches the frontend, or drop the `html` `_type` entirely if nothing legitimately needs raw HTML.

**4. Test suite gives a fundamentally misleading signal**
`pytest tests/` reports **22 failed / 17 passed**. `tests/conftest.py`'s Odoo mock is superficial — `self.env['model'].create()` returns a bare unconfigured `MagicMock`, not something reflecting the input — so tests written against a real `TransactionCase` fail en masse regardless of code correctness. Running the SAME tests via Odoo's real test runner (`--test-enable --test-tags /mcp_gateway`) gives **36 passed / 1 failed / 2 errors of 39** — the true signal, and it's genuinely good. No Makefile/CI/README documents the real command, so nobody appears to run it regularly — which is exactly how the 3 real bugs below (and #1 above) slipped through despite dedicated tests existing.
*Fix:* document `--test-enable --test-tags /mcp_gateway` as canonical; either fix `conftest.py`'s mock to echo `create()` inputs or relabel it "smoke test only, not authoritative."

**5. Webhook ORM auto-trigger is advertised but not implemented**
`mcp.webhook.trigger` has `trigger_on`/`trigger_fields`/`domain` fields, but no `create()`/`write()` override on target models actually fires them — only the HTTP endpoint path works. Configuring an ORM-based trigger silently does nothing.
*Fix:* wire it up (~50 lines per existing HANDOFF.md note), or remove the fields/UI until real.

### Medium

**6. Bulk create/update skip value coercion**
`_dispatch_create_records`/`_dispatch_update_records` (`dispatcher.py:473-510`) pass values straight to the ORM, bypassing `_prepare_create_values()` (handles the `detailed_type→type` rename, date parsing, m2m Command-wrapping) that the singular `create_record`/`update_record` paths use. The same AI-generated payload can succeed via `create_record` and fail via `create_records`.
*Fix:* route each row through `_prepare_create_values` in the bulk path too.

**7. Three genuinely stale/broken tests (found via the real test runner)**
- `test_gateway_run_basic` (`tests/test_gateway.py`) — asserts the raw reply string; the gateway now wraps plain replies as `{"_type":"text","content":"..."}` (structured-output fix, 2026-06-30). Stale assertion.
- `TestGeminiProvider.test_gemini_call_success` (`tests/test_providers.py`) — still patches `google.generativeai`, which no longer exists post-SDK-migration. This is *why* bug #1 was never caught.
- `test_search_read_rejects_non_stored_field_domain` (`tests/test_tools.py`) — creates an `mcp.tool` named `'search_read'`, colliding with the real seeded tool of that name in `default_tools.xml` → `UniqueViolation` against a populated DB.
*Fix:* update the first two; rename the third test's tool to something like `test_domain_check_tool`.

**8. `execute_orm` missing the serialization guard every sibling method has**
`dispatcher.py:756` returns `eval_context['_orm_fn']()` directly, no `_make_serializable()` — unlike `read_group`/`search_read`/`execute_method`. It's the method most likely to return a raw recordset/tuple/datetime, since the AI writes arbitrary ORM code for it. Caught by the outer try/except (confusing generic error, not a hard crash) but breaks the pattern.

**9. Crash — accounting summary compares date to string**
`_dispatch_accounting_health_summary` (`dispatcher.py`, ~541-571): `r['invoice_date_due'] < today` compares a `date` object to `date.today().isoformat()` (a string) — `TypeError` in Python 3, on almost any real invoice with a due date set.
*Fix:* compare against `date.today()` directly.

**10. Crash — missing serialization on attachments**
`_dispatch_get_attachments` (`dispatcher.py:381-390`): `create_date` comes back as a raw `datetime`, never passed through `_make_serializable` — `json.dumps` raises `TypeError` on essentially every call.
*Fix:* wrap the result before returning.

**11. `execute_orm` tool description doesn't mandate `get_model_schema`**
Every other write tool (`create_record`, `create_records`, `update_record`, `update_records`, `import_from_file`) explicitly mandates calling `get_model_schema` first in `data/default_tools.xml`. `execute_orm` doesn't, despite being more dangerous (arbitrary ORM code, bypasses `_prepare_create_values` entirely).

**12. Stdio MCP subprocess has no read timeout**
`gateway.py` ~2148, ~2196: a hung or slow external MCP server blocks the Odoo worker indefinitely on `proc.stdout.readline()`. Since that call never returns, the `finally: proc.kill()` cleanup never runs either — the process leaks.
*Fix:* add a timeout to the read loop.

### Low

**13. `model_selection` can desync from `model_name`**
`models/mcp_agent.py:493` — when a live-fetched model isn't in the pre-seeded `mcp.model.option` table, `model_name` (used for real API calls) is set correctly, but `model_selection` (the dropdown) silently ends up empty.

**14. `.gitignore` doesn't actually ignore `.agents`/`.agy`**
Both are symlinks to the vault; git's dir-only (trailing-slash) gitignore patterns don't match symlinks even when the target is a directory.
*Fix:* drop the trailing slash (`.agents`, `.agy`, not `.agents/`, `.agy/`).

**15. Orphaned scripts reference a deleted model**
`scripts/list_issues.py:27`, `scripts/triage_issues.py:14` still reference `mcp.issue`, removed this branch (migration drops the table). Both will crash if run.

**16. `group_mcp_user` doesn't imply `base.group_user`**
`mcp_session_user_rule` keys off `base.group_user`, not the module's own `group_mcp_user`. Fails *closed* today only because Odoo's normal user-creation flow always includes `base.group_user` — not currently exploitable, but a latent gap if roles are ever assigned programmatically.
*Fix:* add `implied_ids` for robustness.

## Improvements (not bugs)

- **Provider message-formatting is a growing if/elif chain in `gateway.py`**, duplicated in two places (attach-tool_calls, append-tool_result) — exactly the shape that produced bug #1. Consider moving formatting responsibility into each provider adapter instead of a central conditional, so adding a provider can't silently miss a branch again.
- **This session's own session-search** runs an unbounded `searchRead` (ilike join on message content) on every keystroke after debounce, no row cap — unlike `loadRecentSessions`'s `limit: 20`. Add a cap (e.g. 50).
- **Repo hygiene**: `requirements.txt` is legitimate project documentation and should be committed; `scratch/` (throwaway debug scripts + test `.xlsx` fixtures) and `.cursorrules`/`GEMINI.md` (personal AI-tool config) should be gitignored.
- Minor duplication: `execute_orm` and `create_echart` both hand-roll the same "indent, wrap in `def _fn()`, `safe_eval(mode='exec')`" pattern — low priority, 2 call sites.
- `opencode.py`'s `tool_specs` param has a default while the other 5 providers require it positionally — harmless, worth normalizing.
- `sudo_execute` on custom tools is a real privilege-escalation surface by design — appropriately gated to manager/admin-only tool creation today, worth remembering when reviewing custom tools.
- Feature backlog, all code-complete but never browser-verified (no headless browser in this environment): Grok with a live key, prompt-template UI, stdio MCP against a real server, RBAC end-to-end, and this session's search/kebab-menu work.
